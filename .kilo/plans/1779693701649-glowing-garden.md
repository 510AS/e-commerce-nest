## 1. Critical Issues (Must Fix)

### 1.1 Zero Test Coverage
**Severity: CRITICAL** — No `.spec.ts` files exist anywhere in the project. Jest is configured, `@nestjs/testing` and `supertest` are installed, but zero tests have been written for any of the 39 modules.

**Impact:** Regression risk is extremely high. Any refactoring or new feature can silently break existing functionality.

**Recommended actions:**
- Write unit tests for core services first: Auth, Users, Products, Cart, Orders, Payments
- Use `@nestjs/testing` `Test.createTestingModule` pattern
- Mock `PrismaService` with a factory helper
- Target 80% coverage on Phase 1-2 modules before any further feature work

## 2. Security Gaps

### 2.1 No Rate Limiting
**Severity: HIGH** — `@nestjs/throttler` is not installed. No rate limiting on any endpoint. The plan specifies: public 100/min, customer 300/min, vendor/admin 1000/min.

**Fix:**
```bash
npm install @nestjs/throttler
```
Apply `@Throttle()` or `ThrottlerGuard` globally with tiered limits.

### 2.2 Helmet Not Enabled
**Severity: HIGH** — `helmet` is in dependencies but never applied in `main.ts`. Missing security headers (CSP, X-Frame-Options, X-XSS-Protection, etc.).

**Fix:** Add `app.use(helmet())` in `main.ts` bootstrap.

### 2.3 CORS Not Enabled
**Severity: MEDIUM** — CORS origin is configured in `app.config.ts` but `app.enableCors()` is never called in `main.ts`. All cross-origin requests will be blocked by browsers.

**Fix:** Add `app.enableCors({ origin: configService.get('app.corsOrigin') })` in bootstrap.

### 2.4 No Input Sanitization
**Severity: MEDIUM** — No XSS sanitization on user input. `class-validator` validates structure but doesn't sanitize content. User-generated content (product descriptions, reviews, profile fields) is stored raw.

**Fix:** Add a sanitization pipe or middleware before storage.

## 3. Code Quality Issues

### 3.1 Excessive `any` Type Usage (59 instances)
**Severity: WARNING** — Spread across ~25 files. Most common patterns:
- Filter query builder `where` objects declared as `any`
- Prisma enum coercion with `as any`
- Social auth request params typed as `any`
- Promotion/SEO schema builders as `any`

**Fix:** Replace with proper Prisma-generated types:
- `Prisma.ProductWhereInput` instead of `any` for where clauses
- `Prisma.VendorStatus`, `Prisma.OrderStatus` etc. instead of `as any` casts

### 3.2 Missing Error Handling (only 6 try-catch blocks)
**Severity: WARNING** — Most services have no try-catch. Unhandled Prisma errors (unique constraint, foreign key, not found) will crash with 500s instead of proper HTTP error responses.

**Fix:** The global `HttpExceptionFilter` only catches `HttpException`, not raw Prisma errors. Add a Prisma exception filter:
```
@Catch(Prisma.PrismaClientKnownRequestError)
```

### 3.3 Duplicate Vendor Modules
**Severity: WARNING** — Two vendor modules exist:
- `src/modules/vendors/` (stub, used by import path in some files)
- `src/modules/marketplace/vendors/` (full implementation)

AppModule imports `src/modules/vendors/vendors.module` but the full implementation is in marketplace/vendors. This creates confusion and possible runtime issues.

**Fix:** Consolidate into `src/modules/marketplace/vendors/`. Update all import paths. Remove the stub module.

### 3.4 No `shared/` Infrastructure
**Severity: WARNING** — The plan specified `src/shared/` with event-bus, cache, queue, logger, health — none exist. The `@nestjs/bullmq` and `@nestjs/schedule` packages are installed but unused.

**Fix:** Create `src/shared/` with:
- `cache/` — Redis cache service for product details, category tree, pricing
- `queue/` — BullMQ workers for email sending, image processing
- `event-bus/` — Centralized event emitter (currently only in notifications module)

### 3.5 No Events Emitted from Core Services
**Severity: WARNING** — NotificationsService listens for `order.created`, `order.status_changed`, and `checkout.abandoned` events, but NO service emits these events. The notification system is completely disconnected.

**Fix:** Inject `EventEmitter2` into `OrdersService`, `CheckoutService`, and emit events after key actions.

### 3.6 `main.ts` Missing Middleware
**Severity: MEDIUM** — Several packages installed but not wired in `main.ts`:
- `compression` — response compression not enabled
- `cookie-parser` — cookie parsing not enabled
- Helmet not applied (see 2.2)
- CORS not enabled (see 2.3)
- Global prefix `api/v1` from config not applied

## 4. Schema Improvements

### 4.1 Missing Cascade Deletes
**Severity: HIGH** — The following relations have no `onDelete: Cascade`, causing referential integrity errors on delete:
- `Payment → Order` (if order is deleted, payment is orphaned)
- `Shipment → Order` (same)
- `Category → CommissionRule`
- `BusinessAccount → User`
- `PurchaseOrder → BusinessAccount`
- `QuoteRequest → BusinessAccount`

### 4.2 Missing Compound Indexes
**Severity: MEDIUM** — The plan specifies composite indexes on `(ownerType, status, categoryId)` for Product and `(vendorId, status)` for vendor products. These are missing.

### 4.3 `Product.ownerType` + `Product.status` Composite Index
Add: `@@index([ownerType, status])` for marketplace product queries.

## 5. Architecture Gaps

### 5.1 No API Versioning
**Severity: MEDIUM** — The plan specifies `/api/v1/...` URL versioning but `main.ts` applies no global prefix. All endpoints are at the root.

**Fix:** `app.setGlobalPrefix(configService.get('app.apiPrefix', 'api/v1'))`.

### 5.2 No Cursor Pagination
**Severity: MEDIUM** — Only offset-based pagination (page/limit). Cursor pagination is needed for product listings and order history (specified in plan).

### 5.3 Missing Platform/Vendor Guards
**Severity: MEDIUM** — The plan specifies `@PlatformOnly` and `@VendorGuard` for ownership-aware access control. Only JWT + Roles guards exist. Vendor isolation is not enforced.

### 5.4 No Swagger Grouping/Organization
**Severity: LOW** — 39+ modules produce a very long Swagger doc. Consider grouping by domain with `@ApiTags`.

## 7. Performance Optimizations

### 7.1 Redis Caching (Not Implemented)
**Severity: MEDIUM** — Redis is running but no caching layer exists. Hot paths needing caching:
- Product details (TTL: 5min)
- Category tree (TTL: 30min)
- Pricing (TTL: 15min)
- Cart (guest carts in Redis instead of DB)

### 7.2 No Database Connection Pooling Configuration
**Severity: LOW** — PrismaPg adapter uses Pool defaults. No explicit pool configuration for production scale.

### 7.3 N+1 Query Risk
**Severity: LOW** — Several services fetch relations without `include` or batch queries. Cart service, checkout service, and order creation query multiple tables sequentially.
